SIMD-accelerated **integer dot products over byte slices** â€” the
multiply-accumulate at the heart of **quantized machine-learning inference**,
where embeddings and weights are stored as 8-bit integers to cut memory
bandwidth and exploit per-byte vector throughput. The signed `int8`, unsigned
`uint8`, and mixed `uint8Ă—int8` contractions, each summed exactly into a 32-bit
accumulator.

Pure Go: `CGO_ENABLED=0`, stable Go, no `GOEXPERIMENT`. The hot MAC loops are
real integer SIMD on five of the six 64-bit Go targets (see
[Architecture support](#architecture-support)), assembly generated by
[go-asmgen](https://github.com/go-asmgen/asmgen) â€” on a plain `go build`. It is
the integer counterpart to [go-simd/floats](https://github.com/go-simd/floats).

```go

du := int8dot.DotUint8(a, b)   // []uint8 Â· []uint8  -> uint32
```

### Why three functions

INT8 quantization schemes differ in the sign of each operand. Symmetric weights
are `int8`; ReLU-style activations are non-negative and quantize to `uint8`. The
**`uint8 Ă— int8`** layout (`DotU8S8`) is what hardware INT8 GEMM paths
(x86 VNNI, ARM dot-product) are built around, so it gets its own entry point.

### Saturation safety

Each byteĂ—byte product fits in 16 bits, and the sum accumulates in a 32-bit
integer, so there is no clamping or saturation: the result is **exact**. With
worst-case magnitudes `int32` does not overflow until ~131k elements â€” embedding
vectors (256â€“4096 dims of small quantized values) stay comfortably inside. Every
SIMD kernel is required to be **bit-for-bit identical** to the scalar reference
on every architecture (differential tests + `FuzzDot`).

## How it works

Integer multiply-accumulate is associative and exact, so â€” unlike the
floating-point sibling â€” there is no lane-order or ULP subtlety: the kernels are
required to equal a straight-line scalar loop bit for bit. Each kernel widens
byte products to 16 bits, accumulates them into several 32-bit lanes (for
instruction-level parallelism), and horizontally sums at the end, with a scalar
tail for the sub-stride remainder.

| arch | int8 / uint8 kernel | u8Ă—s8 kernel | notes |
|---|---|---|---|
| **amd64**   | `VPMOVSXBW/VPMOVZXBW` + `VPMADDWD` (AVX2) | AVX2 | runtime-gated on AVX2; VNNI deferred (see below) |
| **arm64**   | `VSMULL/VUMULL` widening (NEON, **Go 1.27+**) | `VUXTL`+`VSXTL`+`VSMULL` (NEON, **Go 1.27+**) | scalar on stable Go â‰¤ 1.26 (see below) |
| **ppc64le** | `VMULESB/VMULOSB` even/odd widening (VMX) | *scalar* | no mixed-sign byte multiply on Power |
| **s390x**   | `VMEB/VMOB` even/odd widening (vector facility) | *scalar* | big-endian; no mixed-sign byte multiply |
| **riscv64** | `VWMULVV` + `VWREDSUMVS` (RVV) | `VWMULSUVV` (RVV) | runtime-gated on RVV |
| **loong64** | `VMULWEVHB/VMULWODHB` even/odd widening (LSX) | *scalar* | no mixed-sign byte multiply on LoongArch |

Every other architecture uses the portable scalar reference.

### Architecture support

This package targets **stable Go**. A few wrinkles in the current assembler shape
the table above; all are honest, documented trade-offs rather than silent gaps:

- **arm64 â€” NEON SIMD on Go 1.27+, scalar on stable Go â‰¤ 1.26.** All three dot
  products have a NEON widening multiply-accumulate kernel (`VSXTL`/`VUXTL` widen
  the bytes to 16-bit, `VSMULL`/`VUMULL` widen-multiply to 32-bit, `VADD`
  accumulates, `VADDV` reduces, with a scalar tail). Those integer-multiply
  mnemonics are only exposed by the Go arm64 assembler from **Go 1.27** (1.26 only
  has `VPMULL`, polynomial multiply, useless for integer arithmetic), so the
  kernel is `//go:build arm64 && go1.27` and **stable Go falls back to the scalar
  reference** â€” there is no SIMD on Go â‰¤ 1.26. `DotU8S8` is full SIMD here too
  (unlike ppc64le/s390x/loong64): the unsigned operand is zero-extended, the
  signed operand sign-extended, and the signed halfword multiply then yields the
  exact mixed-sign product.

  The ARMv8.2 dot-product instructions (`SDOT`/`UDOT`/`USDOT`) would be the
  one-instruction kernel, but as of the Go 1.27 dev tree the assembler exposes
  them only in their **SVE** form (`ZSDOT`/`ZUDOT`/`ZUSDOT`, scalable Z-registers
  + predicate) â€” there is no NEON/Advanced-SIMD `SDOT` mnemonic, and SVE is not
  implemented by Apple Silicon (where this package is developed and natively
  tested). The portable NEON widening-multiply path is used instead; a `ZSDOT`
  kernel is a clean follow-up once the assembler grows a NEON `SDOT` (or a
  validated SVE runner exists).
- **amd64 â€” AVX2, not VNNI (yet).** `VPDPBUSD` (AVX-512-VNNI) is the ideal
  `uint8Ă—int8` instruction, but it cannot be validated or coverage-gated in this
  project's CI: qemu's TCG implements no AVX-512, so neither the differential
  tests nor the 100% gate can exercise it. Rather than ship an unvalidated
  kernel, amd64 uses the AVX2 path (validated bit-exact on a native x86-64 VM). A
  `VPDPBUSD` fast path is a clean follow-up once a VNNI-capable, test-covered
  runner is available.
- **ppc64le / s390x / loong64 â€” `DotU8S8` is scalar.** None of these ISAs has a
  mixed unsignedĂ—signed byte multiply primitive; synthesising one would cost more
  than the scalar loop. `Dot` and `DotUint8` are full SIMD on all three.

## Validation

Each SIMD kernel is checked **bit-exact against the scalar reference** â€” there is
a single right answer for integer MAC â€” on its real target:

- **amd64** natively on an x86-64 VM (AVX2), **arm64** natively;
- **ppc64le** natively on real POWER10 silicon (GCC Compile Farm, VSX, Go
  1.26.4), and **s390x, riscv64, loong64** under qemu (`tonistiigi/binfmt`), with
  the big-endian s390x run specifically pinning byte/lane order.

Beyond the six SIMD targets, the portable scalar reference is now build- and
test-validated on a **seventh architecture, ppc64 (big-endian)**, on real POWER9
silicon (GCC Compile Farm) â€” bit-exact on a big-endian target distinct from
s390x's vector kernel. ppc64 BE carries no VSX build tag, so it exercises the
generic path. Framing: **six SIMD targets, validated on seven architectures.**

`FuzzDot` (a differential fuzzer over all three operations) and a size sweep
covering the vector body, the scalar tail and the boundaries pass on every
architecture, at **100% statement coverage** enforced in CI.

### Missing-mnemonic findings

While building this, several instructions turned out to be absent from the
**stable Go 1.26** assembler (present only in the Go 1.27 dev tree), or to behave
incorrectly under emulation â€” recorded here for the next person:

- **arm64:** the integer vector multiplies `VSMULL/VUMULL/VSMLAL/VUMLAL` and the
  widening `VSXTL/VUXTL` are **absent from stable Go 1.26** and arrive in **Go
  1.27** (1.26 has only the polynomial `VPMULL`). The dot-product instructions
  `SDOT/UDOT/USDOT` are in Go 1.27 **only as SVE** (`ZSDOT/ZUDOT/ZUSDOT`, Z-regs +
  predicate) â€” there is still **no NEON/Advanced-SIMD `SDOT`** mnemonic in
  `cmd/internal/obj/arm64`, so the kernel uses the NEON widening-multiply path
  (which also runs on Apple Silicon, where SVE is unavailable).
- **ppc64le:** no `VMSUMMBM/VMSUMUBM` (byte multiply-sum); only `VMSUMUDM`
  (doubleword). The even/odd `VMULESB/VMULOSB`â†’`VMULESH/VMULOSH` path is used
  instead.
- **loong64:** the unsigned widening *add* `VADDWEV.W.HU` is **mis-emulated by
  qemu-la464** (corrupts the high lanes); the unsigned kernel widens with the
  multiply-add `VMADDWEV.W.HU` against a ones vector instead, which qemu handles
  correctly.
